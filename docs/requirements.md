# 仮想オーディオルーティングアプリ 定義書

**コードネーム:** 仮称「Conflux」(変更可)
**対象 OS:** macOS 14.4 以降(Apple Silicon / Intel)
**版:** v0.1（ドラフト）
**位置づけ:** BlackHole の課題を解決する、ルーティング＋キャプチャ統合型オーディオユーティリティ（Loopback 対抗）

---

## 1. 目的と背景

### 1.1 解決する課題
BlackHole に代表される既存の仮想オーディオドライバは「素通しの仮想ケーブル」に過ぎず、実運用では以下が負担になっている。

| # | BlackHole の課題 | 影響 |
|---|---|---|
| C1 | Audio MIDI Setup での Aggregate / Multi-Output 手動構築 | セットアップが最大の脱落要因 |
| C2 | アプリ単位のソース分離が不可（システム全体素通し） | 「Spotify だけ」が録れない |
| C3 | モニタリング不可（Multi-Output を別途構築しないと聞けない） | 配信・録音時に自分の音が確認できない |
| C4 | クロックドリフト / サンプルレート不整合 | 散発的なクリックノイズ・同期ずれ |
| C5 | Multi-Output / Aggregate の音量制御不可（macOS 仕様） | ミックスバランスが取れない |
| C6 | メータリング等の可視化が皆無 | 状態がブラックボックス |
| C7 | GPL-3 ライセンス | 商用・クローズド配布に制約 |

### 1.2 本製品の方針
- **キャプチャ用途は Core Audio Process Tap（macOS 14.4+）で実現**し、ユーザーに Audio MIDI Setup を一切触らせない。
- **ルーティング用途（他アプリの入力デバイス化）は自作 AudioServerPlugIn 仮想デバイスで実現**する。
- 両者を **App 内ミキサー**で束ね、SwiftUI 制御アプリが裏で private aggregate を自動生成する **ハイブリッド構成**を採る。
- ドライバは **Apple サンプルから起こし**、GPL（BlackHole 由来）を回避してライセンス自由度を確保する。

## 2. スコープ

### 2.1 v1 スコープ内
- 任意プロセス／システム全体の音声キャプチャ（Tap）
- マイク・オーディオIF など実入力デバイスの取り込み
- ソース → バス → 宛先のルーティング行列
- 自作仮想デバイス（固定 16ch、複数インスタンス）への出力
- リアルタイムモニタリング（実出力へのパススルー）
- ソース単位の gain / mute / pan
- チャンネル単位ライブメーター
- ファイル録音（最終ミックス or バス単位）
- メニューバー常駐 UI

### 2.2 v1 スコープ外（将来）
- ドライバ内ミキシング（driver-as-mixer、超低レイテンシ）
- 仮想デバイスのチャンネル数ライブ可変
- VST/AU エフェクト挿入
- iOS / iPadOS 対応
- Windows 対応

## 3. 用語定義

| 用語 | 定義 |
|---|---|
| AudioServerPlugIn | coreaudiod がロードする HAL プラグイン。`/Library/Audio/Plug-Ins/HAL/` に配置する `.driver` バンドル。本製品の仮想デバイス実体 |
| Process Tap | `AudioHardwareCreateProcessTap` / `CATapDescription` による、特定プロセス群・デバイス出力のキャプチャ機構（macOS 14.4+） |
| 仮想デバイス | 本製品が publish するループバック専用デバイス。output に書かれた音を input へ返す |
| App エンジン | 制御アプリ内のオーディオミキサー。本製品の頭脳 |
| バス | ルーティングの論理出力単位。1 仮想デバイス＝1 バスに対応 |
| 制御プレーン | App→driver の設定伝達経路。カスタム HAL プロパティを使用 |
| ドリフト補正 | 異クロック間の累積ずれを非同期 SRC 比で吸収する処理 |

## 4. システム構成

### 4.1 レイヤー構成
```
[ソースapp] --Tap--> ┌─────────────────────────┐
[マイク/IF] --------> │   App オーディオエンジン   │
                     │  gain/mix/route/meter    │
                     └──┬───────────┬───────────┘
        モニタリング ←──┘           │ 最終ミックス書込み
      (実出力デバイス)              ▼
                          [仮想デバイス output]
                                 │ loopback(driver内)
                                 ▼
                          [仮想デバイス input] ──> [Zoom/OBS/録音app]

制御: SwiftUI UI ──カスタムHALプロパティ──> driver（デバイス数/ch レイアウト）
```

### 4.2 設計原則:driver は薄く、App をミキサーにする（App-as-mixer）
- **仮想デバイス（driver）**は純ループバックバスのみ。賢い処理はゼロ。
- **App エンジン**が頭脳。Tap＋実入力を取り込み、ソース単位の gain/mute/pan を当て、ルーティング行列に従ってバスごとに合算 → 各仮想デバイス output へ書き込む。同時にモニタリング・メーター・録音を担う。
- driver-as-mixer は低レイテンシだが realtime セーフなミキシングと共有メモリ IPC のコストが高く、v1 では採用しない。App-as-mixer の追加遅延（数 ms）はモニタリング以外で許容する。

## 5. 機能要件

### 5.1 仮想デバイス（driver）
- F-D1: Audio MIDI Setup・各種オーディオアプリに input/output 両用デバイスとして表示される。
- F-D2: 16ch / 32-bit float をサポートする（Core Audio ネイティブ整合）。
- F-D3: output に書かれたサンプルをロックフリーリングバッファ経由で input に返す。
- F-D4: 複数インスタンス（複数バス）を publish できる。
- F-D5: 正確な `GetZeroTimeStamp` を提供し、クロック源として安定動作する。
- F-D6: カスタム HAL プロパティ経由で、App からデバイス数・チャンネルマップ・ルーティングを受領する。

### 5.2 キャプチャ（Tap）
- F-C1: 特定プロセス群／システム全体の出力をタップできる。
- F-C2: 初回タップ時に TCC 許可プロンプト（`NSAudioCaptureUsageDescription`）を提示する。
- F-C3: タップは private aggregate（実出力デバイスを main sub-device、タップを sub-tap、AutoStart 有効）として構築する。
- F-C4: 取得は `AudioDeviceCreateIOProcIDWithBlock` を直接使用する（AVAudioEngine の retarget は不可のため不採用）。
- F-C5: 出力デバイスのステレオペア数に比例する減衰を補正する。
- F-C6: 無音化バグ（長時間後の全サンプル 0.0f）を検知し、Tap・aggregate を完全再生成して自動復旧する。

### 5.3 ルーティング・ミキシング（App エンジン）
- F-E1: ソース（Tap / 実入力）× バスのルーティング行列を保持する。
- F-E2: ソース単位の gain / mute / pan を realtime セーフに反映する。
- F-E3: バスごとに合算し、対応する仮想デバイス output へ書き込む。
- F-E4: チャンネル単位のピーク／RMS メーターを供給する。
- F-E5: 最終ミックスまたは任意バスをファイルへ録音できる。

### 5.4 モニタリング
- F-M1: 任意バスを実出力デバイスへパススルーできる（オン/オフ）。
- F-M2: モニタリング経路は最短レイテンシ構成とする。

### 5.5 UI / UX
- F-U1: ノードグラフ型のルーティング表示（ソース → バス → 宛先）。
- F-U2: 「録る/流す」をワンクリックで開始し、裏で aggregate を自動構築する。
- F-U3: 実行中プロセス一覧からソースを選択できる。
- F-U4: チャンネル別ライブメーターを常時表示する。
- F-U5: アプリ／チャンネル単位ゲイン操作（C5 の音量制御課題を解決）。
- F-U6: メニューバー常駐、クイックアクセスを提供する。

## 6. 非機能要件

### 6.1 リアルタイム性・性能
- NF-P1: driver の IO コールバックは無確保（malloc 禁止）・無ロック・無 syscall とする。
- NF-P2: 設定変更は SPSC ロックフリーコマンドキュー＋アトミックなパラメータスナップショットで音声スレッドへ渡す。
- NF-P3: メーター・ゲイン等の DSP は vDSP を用い realtime スレッド外で処理する。
- NF-P4: 16ch では SoA メモリレイアウトを採用する。
- NF-P5: App-as-mixer 経路の追加遅延は数 ms 以内を目標とする。

### 6.2 クロック・同期
- NF-S1: キャプチャ側（実出力＋Tap）は単一 aggregate に同居させ同一クロックに揃える。
- NF-S2: 非同期 SRC は「App エンジン → 仮想デバイス」の 1 境界のみに集約する。
- NF-S3: 当該境界のリングバッファに PI 制御のドリフト推定器を置き、SRC 比を動的調整してアンダーラン／オーバーランを防ぐ。

### 6.3 信頼性
- NF-R1: Tap 無音化を RMS ＋ running 状態のウォッチドッグで検知する（無音と区別するため両条件を併用）。
- NF-R2: 復旧時のバッファ再構築はクロスフェードで行い、consumer 側にグリッチを出さない。
- NF-R3: 出力デバイス切替・サンプルレート再交渉時も自動再構成する。

### 6.4 セキュリティ・権限
- NF-SE1: Tap 利用は TCC 許可必須。許可状態を事前確認し、UI で誘導する。
- NF-SE2: driver バンドルは最低 ad-hoc 署名、配布時は Developer ID 署名＋notarization。
- NF-SE3: aggregate は `kAudioAggregateDeviceIsPrivateKey = true` でグローバル露出を避ける。

### 6.5 互換性
- NF-CO1: Apple Silicon / Intel 両対応（ユニバーサル）。
- NF-CO2: サポート最小 OS は macOS 14.4。
- NF-CO3: 主要 DAW・配信ソフト（OBS, Zoom, GarageBand 等）から入力デバイスとして選択可能であること。

## 7. 技術的制約・既知リスク

| ID | リスク | 影響 | 対策 |
|---|---|---|---|
| R1 | Tap が長時間後に全サンプル 0.0f になる（IOProc は継続発火、無音と判別困難） | キャプチャ停止 | ウォッチドッグ検知＋Tap/aggregate 完全再生成＋クロスフェード（F-C6, NF-R1/R2） |
| R2 | 出力デバイスのステレオペア数に比例した減衰（例:8out で −12dB） | レベル不正確 | ペア数からの補正ゲイン（F-C5） |
| R3 | AVAudioEngine はタップ aggregate に retarget 不可（noErr を返すが既定入力を読む） | 実装誤り | `AudioDeviceCreateIOProcIDWithBlock` を直接使用（F-C4） |
| R4 | 仮想デバイスのチャンネル数ライブ変更はクライアントを切断 | 中断 | v1 は固定 16/32ch を publish し内部ルーティングで吸収 |
| R5 | coreaudiod へ XPC を通しにくい | 制御不能 | カスタム HAL プロパティセレクタで設定 push（第8章） |
| R6 | 異クロック混在によるドリフト | クリックノイズ | SRC を 1 境界へ集約＋PI ドリフト補正（NF-S1〜S3） |

## 8. 制御プレーン（App ↔ driver）

- カスタム HAL プロパティセレクタを定義する。
- App は `AudioObjectSetPropertyData` で設定（デバイス数・チャンネルマップ・各バスのルーティング）を push する。
- プラグインは `SetPropertyData` で受領し、内部状態へ realtime セーフに反映する。
- これは Loopback 系製品の標準的手法に準ずる。

## 9. 開発フェーズ計画

リスクの高いプリミティブから検証する。

### Phase 0 — 危険プリミティブのスパイク
- 0(a): 最小 AudioServerPlugIn（16ch ループバック）が Audio MIDI Setup に表示され、App から書込み→読出しできる。
- 0(b): 最小 Tap キャプチャ（システム音をファイルへ）＋無音ウォッチドッグ＋減衰補正。

### Phase 1 — エンジン
- Tap＋実入力を App 内ミキサーで合算し仮想デバイスへ送る。
- クロック橋渡し（SRC＋ドリフト補正）を完成させる。

### Phase 2 — 制御プレーン
- カスタム HAL プロパティ、複数仮想デバイス対応。

### Phase 3 — UI
- ルーティンググラフ、メーター、ソース別 gain、モニタリング、メニューバー。

### Phase 4 — 配布
- 署名・notarization、インストーラ（HAL 配置＋`coreaudiod` kickstart）、TCC プロンプト整備。

## 10. ライセンス・配布要件
- driver は Apple サンプルから起こし、GPL（BlackHole 由来）を回避する。
- 商用・クローズド配布を前提にライセンスを設計する。

## 11. 受け入れ基準（フェーズ別）

| フェーズ | 受け入れ基準 |
|---|---|
| Phase 0(a) | 仮想デバイスが Audio MIDI Setup に 16ch で出現し、App が output へ書いた信号を input から bit 一致で読み出せる |
| Phase 0(b) | 任意プロセスの音をファイル録音でき、24h 連続でも無音化せず（ウォッチドッグ作動を含め）録れ続ける。減衰補正後のレベル誤差 ±0.5dB 以内 |
| Phase 1 | 「Spotify → 本アプリ → 仮想デバイス → QuickTime」で録音でき、10 分間クリックノイズ・ドリフト破綻ゼロ |
| Phase 2 | App から複数バスを動的に生成・ルーティング変更でき、音声スレッドにグリッチが出ない |
| Phase 3 | 非開発者が説明書なしでソース選択→録音→モニタリングまで到達できる |
| Phase 4 | 署名・notarization 済みインストーラで配布でき、クリーン環境でインストール→TCC 許可→動作する |
